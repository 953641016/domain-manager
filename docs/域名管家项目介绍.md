# 域名管家小白手册

> 适用对象：第一次听说本项目的人、业务负责人、域名管理人员、运维人员、准备评估是否需要引入本系统的人。
>
> 阅读目标：不需要懂代码，也能明白这个项目解决什么问题、谁会用、怎么流转、适不适合自己的团队。

---

## 1. 一句话介绍

域名管家是一个企业内部使用的域名和 DNS 自动化管理系统。

它让业务同事可以在飞书里提交域名注册、DNS 解析等申请；域名专员在飞书里审批和执行；Web 后台负责账号配置、记录查询、统计、到期提醒和审计日志。

简单说：

> 以前靠人手动买域名、改解析、查记录、追审批；现在把这些流程放进系统，由飞书发起和确认，由后台沉淀记录，由系统对接 Cloudflare、GoDaddy 等平台执行。

---

## 2. 它主要解决什么问题

很多公司在做多个产品、网站、落地页、海外项目时，域名会越来越多。常见问题包括：

1. 域名分散在不同平台，比如 Cloudflare、GoDaddy、DNSPod 等。
2. 业务同事要买域名或改 DNS，经常在飞书、群聊、表格里临时找人。
3. 谁申请的、谁审批的、谁执行的，经常没有完整记录。
4. DNS 解析改错后，很难快速追查是谁改的、什么时候改的。
5. 域名快过期时没人提醒，容易造成业务中断。
6. API Key、注册商账号、DNS 账号集中在少数人手里，不好授权，也不安全。
7. 业务人员不应该直接登录 Cloudflare / GoDaddy，但又经常需要发起域名相关需求。

域名管家的核心价值，就是把这些零散、靠人工记忆和聊天推进的流程，变成可审批、可执行、可追踪、可审计的标准流程。

---

## 3. 适合谁使用

这个系统适合下面这些团队：

| 团队情况 | 是否适合 |
|---|---|
| 有多个产品、多个网站、多个域名 | 适合 |
| 域名注册和 DNS 解析经常变更 | 适合 |
| 业务同事经常找技术或运维帮忙改解析 | 适合 |
| 公司内部使用飞书协作 | 很适合 |
| 需要审批、权限隔离、操作日志 | 很适合 |
| 域名账号和 API Key 不想直接暴露给业务人员 | 很适合 |
| 只有一两个域名，基本不变更 | 不一定需要 |
| 单人维护，完全不需要审批 | 不一定需要 |
| 不使用飞书，也不准备接入飞书 | 需要改造后才适合 |

判断是否需要这个系统，可以直接问：

1. 你们有没有多个域名、多个注册商账号或多个 DNS 账号？
2. 业务人员是否经常提域名注册或 DNS 修改需求？
3. 现在是否靠群消息、表格、人工登录平台来处理？
4. 是否需要知道每次修改是谁发起、谁审批、谁执行？
5. 是否担心域名过期、解析误改、权限滥用？

如果上面有两个以上回答“是”，这个系统就有价值。

## 4. 这个系统不是普通后台

理解这个项目，最重要的是先理解一句话：

> 飞书负责业务流程，Web 后台负责配置和查询。

也就是说，它不是一个让所有人在网页上随便点按钮买域名、改 DNS 的普通后台。

它分成两层：

| 层级 | 入口 | 负责什么 | 不负责什么 |
|---|---|---|---|
| 业务层 | 飞书文档、飞书机器人、飞书卡片 | 提交申请、审批、确认、执行、通知 | 不做复杂配置 |
| 配置/管理层 | Web 后台 | 用户、角色、注册商账号、DNS 账号、默认配置、记录查询、统计、日志 | 不推进业务流程 |

这样设计的原因是安全。

域名和 DNS 是高风险资产。改错一条 DNS 记录，可能导致网站打不开、邮件收不到、验证失败、广告投放异常。注册商账号和 API Key 也属于敏感配置，不能让所有业务人员直接接触。

所以系统把“业务操作入口”放在飞书，把“配置和审计入口”放在 Web，把关键敏感操作再收口到超级管理员的飞书二次确认。

## 5. 四类用户角色

系统里有四类角色。理解角色后，基本就能理解整套流程。

### 5.1 业务用户 business

业务用户就是普通业务同事。

他们通常不登录 Web 后台，只在飞书里提交需求。

他们能做：

- 提交域名注册申请
- 提交 DNS 解析申请
- 查看自己申请的处理状态

他们不能做：

- 直接登录注册商平台
- 直接修改 DNS
- 直接审批自己的申请
- 看到 API Key 或服务商账号
- 进入 Web 后台管理配置

适合例子：

> 某个产品负责人要上线一个新站点，需要注册 `example.ai`，并加几条解析记录。他不需要知道 Cloudflare 怎么用，只需要在飞书里提交申请。

### 5.2 域名专员 domain_spec

域名专员是负责域名业务的人。

他们能看和管理分配给自己的域名、注册账号、DNS 账号，能审批归属自己的业务用户申请。

他们能做：

- 审批业务用户提交的域名申请
- 审批 DNS 解析申请
- 查看自己负责的域名和 DNS 记录
- 接收到期提醒
- 在飞书里直接发起注册或解析操作
- 在 Web 后台查看自己的记录、统计和日志

他们不能做：

- 管理所有用户
- 修改别的专员负责的账号和域名
- 绕过超管确认去改敏感配置

适合例子：

> 公司有多个业务线，每个域名专员负责一批域名和账号。业务同事发起申请后，由对应专员审批和处理。

### 5.3 系统管理员 admin

系统管理员偏技术运维角色，负责系统本身和用户管理。

当前项目的核心边界是：

> admin 管系统和人员，不碰域名业务。

他们能做：

- 管理用户
- 查看系统日志、审计记录、统计信息
- 维护系统配置
- 处理系统运行问题

他们不能做：

- 直接注册域名
- 直接修改 DNS
- 查看域名账号 API Key
- 参与业务侧审批

这个设计是为了职责分离：技术运维人员不一定应该接触业务域名资产。

### 5.4 超级管理员 super_admin

超级管理员拥有最高权限，也是关键操作的最终确认人。

他们能做：

- 查看全部数据
- 管理全部配置
- 管理用户和角色
- 作为最终确认人，在飞书里确认敏感操作

最重要的是：

> 涉及注册商账号、DNS 账号、用户角色、系统关键配置等敏感操作，即使是超管自己在 Web 发起，也必须到飞书里再次确认，系统才会真正执行。

这样可以防止 Web 后台账号被盗、浏览器会话被劫持、误操作等风险。

## 6. 核心功能模块

### 6.1 飞书登录

系统支持飞书 OAuth 登录。

Web 后台用户通过飞书身份登录，系统根据飞书用户 ID 找到对应的用户记录和角色，再决定能看到哪些页面、能做哪些操作。

### 6.2 域名管理

用于记录和查看公司管理的域名。

典型信息包括：

- 域名名称
- 注册商
- 注册账号
- DNS 服务商
- DNS 账号
- 到期时间
- 负责人
- 当前状态

注意：按照当前设计，Web 主要用于查询和配置，不应该把业务流程推进放在 Web 里。

### 6.3 DNS 解析管理

DNS 解析就是告诉互联网：

> 访问这个域名时，应该指向哪个服务器、哪个服务、哪个验证记录。

常见 DNS 记录包括：

| 类型 | 用途 |
|---|---|
| A | 把域名指向 IPv4 地址 |
| AAAA | 把域名指向 IPv6 地址 |
| CNAME | 把一个域名别名指向另一个域名 |
| MX | 邮件服务器配置 |
| TXT | 文本验证，比如 Google、Cloudflare、邮箱验证 |
| NS | 指定域名使用哪些 DNS 服务器 |
| SRV | 某些服务发现配置 |

业务用户不需要理解所有记录类型，只需要按模板填写需求。域名专员负责判断是否合理。

### 6.4 申请审批

申请审批是系统的核心。

典型流程是：

1. 业务用户在飞书里提交域名注册或 DNS 解析申请。
2. 系统检查用户身份和归属关系。
3. 系统生成申请记录。
4. 系统把申请发送给对应域名专员。
5. 域名专员在飞书卡片里审批。
6. 审批通过后，系统调用 Cloudflare / GoDaddy 等平台接口执行。
7. 系统记录执行结果。
8. 系统通知申请人和相关负责人。
9. Web 后台可以查询整条记录和日志。

### 6.5 注册商和 DNS 服务商集成

系统支持对接外部平台。

当前主要包括：

- Cloudflare
- GoDaddy

Cloudflare 是首选平台，因为它既支持域名注册，也支持 DNS 解析，API 能力比较完整。

系统通过适配器方式对接不同服务商。也就是说，业务流程尽量保持统一，底层执行时再根据注册商或解析商调用对应平台。

### 6.6 到期提醒

系统会定时检查域名到期时间。

到期提醒的价值是避免：

- 域名过期导致网站打不开
- 邮箱、接口、广告落地页受影响
- 域名被别人抢注
- 临时续费找不到负责人

域名专员可以提前收到提醒，及时处理续费或迁移。

### 6.7 审计日志

审计日志就是系统留下的操作记录。

它回答这些问题：

- 谁发起了申请？
- 谁审批了申请？
- 谁确认了敏感操作？
- 操作从哪里发起，Web 还是飞书？
- 改了什么内容？
- 什么时候改的？
- 执行成功还是失败？
- 失败原因是什么？

这对排查问题、权限审计、事故复盘都很重要。

## 7. 典型业务场景

### 场景一：业务同事申请注册新域名

例子：

> 新产品要上线，业务同事想注册 `newproduct.com`。

流程：

1. 业务同事在飞书文档或飞书机器人里填写域名。
2. 系统检查这个用户是否在系统中，是否归属某个域名专员。
3. 系统查询域名是否可注册。
4. 如果可注册，生成申请。
5. 对应域名专员收到飞书审批卡片。
6. 专员确认注册商、注册账号、价格等信息。
7. 专员审批通过。
8. 系统调用注册商接口注册域名。
9. 注册成功后，系统保存域名记录。
10. 系统通知申请人注册结果。

业务同事看到的是：“我提交了申请，后面有人审批，结果会通知我。”

域名专员看到的是：“谁要注册什么域名，是否可注册，用哪个账号执行，执行结果如何。”

### 场景二：业务同事申请新增 DNS 解析

例子：

> 技术同事给了一个服务器 IP，需要把 `api.example.com` 指向这个 IP。

流程：

1. 业务同事在飞书里填写域名、主机记录、记录类型、记录值。
2. 系统创建 DNS 解析申请。
3. 域名专员收到审批卡片。
4. 专员检查记录是否合理。
5. 审批通过后，系统调用 Cloudflare API 添加解析。
6. 系统保存执行结果。
7. 申请人收到通知。

### 场景三：修改敏感配置

例子：

> 域名专员想新增一个 Cloudflare 账号，里面包含 API Key。

流程：

1. 专员在 Web 后台填写账号信息。
2. 系统不直接写入生效。
3. 系统创建一条待确认记录。
4. 超级管理员收到飞书确认卡片。
5. 超管在飞书里确认。
6. 系统才真正保存配置。
7. 系统记录完整审计日志。

这就是“飞书二次确认”。

## 8. Web 后台能做什么

Web 后台适合管理员和域名专员使用。

常见页面和用途：

| 页面 | 用途 |
|---|---|
| Dashboard 仪表盘 | 看整体统计、近期申请、状态分布 |
| 域名列表 | 查询系统管理的域名 |
| 域名详情 | 查看域名基础信息和 DNS 记录 |
| 申请记录 | 查看申请历史、状态、执行结果 |
| 到期提醒 | 查看即将到期的域名 |
| 服务商配置 | 配置 Cloudflare、GoDaddy 等服务商 |
| 注册账号配置 | 配置域名注册账号和 API Key |
| DNS 账号配置 | 配置 DNS 服务商账号和 API Key |
| 默认配置 | 设置默认注册商、默认 DNS 服务商、默认账号 |
| 用户管理 | 添加用户、设置角色、绑定飞书用户 |
| 审计日志 | 查看系统操作记录 |
| 统计报表 | 看申请数量、成功失败情况、处理趋势 |

一句话总结：

> Web 是后台账本和配置台，不是业务人员直接操作域名的平台。

## 9. 飞书里能做什么

飞书是业务流转入口。

主要能力：

- 业务用户提交域名注册申请
- 业务用户提交 DNS 解析申请
- 域名专员审批申请
- 域名专员直接发起注册或解析
- 超级管理员确认敏感操作
- 系统发送成功或失败通知
- 用户查询申请状态

飞书的价值是：

1. 业务用户不用学习 Web 后台。
2. 审批人在日常办公工具里处理，不需要切系统。
3. 关键确认发生在飞书客户端，安全性更高。
4. 所有消息和操作都更贴近公司实际协作方式。

## 10. 权限和安全边界

这个项目对权限要求很严格。

必须记住几条原则：

1. 业务用户只能提交申请，不能直接操作。
2. 域名专员只能管理自己负责的域名和账号。
3. 系统管理员管理用户和系统，不碰域名业务。
4. 超级管理员拥有最高权限，但关键操作仍需飞书二次确认。
5. 注册商账号、DNS 账号、API Key 等敏感数据要加密存储。
6. 新增、修改、删除敏感配置，不能绕过飞书确认。
7. 所有关键操作都要写审计日志。

这套设计不是为了麻烦，而是为了降低事故概率。

域名和 DNS 出问题，影响通常很直接：

- 官网无法访问
- API 无法请求
- 邮件无法收发
- 第三方验证失败
- 广告落地页打不开
- 搜索收录或业务转化受影响

所以系统宁愿流程多一步，也不能让高风险操作变成随手一点。

## 11. 常见名词解释

### 域名

比如 `example.com`、`abc.ai`。用户在浏览器里输入的地址，本质上是互联网资源的名字。

### 注册商

卖域名、管理域名注册信息的平台，比如 Cloudflare、GoDaddy。

### DNS

DNS 负责把域名转换成服务器地址。

可以理解为：

> 域名是名字，DNS 是通讯录。

### DNS 解析记录

告诉 DNS 这个域名应该指向哪里。

例如：

- `www.example.com` 指向某个服务器
- `api.example.com` 指向接口服务器
- `example.com` 添加 Google 验证 TXT

### API Key

系统调用 Cloudflare / GoDaddy 等平台接口时需要的密钥。

它很敏感，不能随便发给别人，也不能写死在代码里。

### 审批

业务用户提交申请后，域名专员判断是否允许执行。

### 超管确认

涉及敏感配置时，系统向超级管理员发送飞书卡片。超管点击确认后，系统才真正执行。

### 审计日志

系统自动记录的操作痕迹，用来追踪和复盘。

## 12. 怎么向完全不了解的人介绍

推荐按这个顺序讲，不要一上来讲技术。

### 第一步：先问现状

可以这样问：

> 你们现在域名和 DNS 是怎么管理的？比如买新域名、加解析、改解析、查到期时间，是谁负责，在哪里记录？

### 第二步：确认痛点

继续问：

> 有没有遇到过申请靠群聊、记录靠表格、改完没人知道、域名快过期才发现、DNS 改错很难追责这种情况？

### 第三步：一句话说明系统

> 这个项目就是把域名注册、DNS 解析、审批、执行、日志、到期提醒整合起来。业务在飞书提需求，专员在飞书审批，系统自动对接 Cloudflare 或 GoDaddy 执行，Web 后台负责配置和查询。

### 第四步：说明价值

> 它的价值不是多一个后台，而是减少人工操作、规范审批、保护账号密钥、避免域名过期和 DNS 误改，并且每一步都有记录。

### 第五步：判断是否需要

> 如果你们域名少、变更少、一个人手工维护就够，可能不急着用。如果你们域名多、业务经常提需求、需要审批和日志，那就很适合。

## 13. 对外介绍话术

### 30 秒版本

域名管家是一个企业内部的域名和 DNS 管理系统。业务同事在飞书里提交域名注册或解析需求，域名专员在飞书审批，系统自动对接 Cloudflare、GoDaddy 等平台执行。Web 后台负责账号配置、域名记录、到期提醒、统计和审计。它适合域名多、DNS 变更频繁、需要权限隔离和审批留痕的团队。

### 1 分钟版本

很多团队管理域名时，都是业务在飞书里找人，技术或运维再手动登录 Cloudflare、GoDaddy 去注册域名或改 DNS。时间久了以后，域名分散、账号分散、审批不规范、记录不完整，还容易忘记续费。域名管家就是把这套流程系统化：业务在飞书提交，专员在飞书审批，系统自动执行并记录结果；Web 后台只做配置、查询、统计和审计。这样既方便业务提交需求，也能保护域名账号和 API Key，出了问题还能查到完整链路。

### 3 分钟版本

这个项目可以理解成公司内部的“域名工单和自动执行平台”。

它不是单纯的域名列表，也不是普通后台。它把域名相关工作拆成两部分：业务流程在飞书里完成，配置和记录在 Web 后台管理。业务同事要注册域名、添加 DNS 解析，不需要登录 Cloudflare 或 GoDaddy，只需要在飞书里提交申请。系统会根据用户身份找到负责的域名专员，专员在飞书里审批，通过后系统自动调用外部平台接口执行。执行结果、失败原因、审批历史、操作人都会写入数据库和审计日志。

对于公司来说，它解决的是三类问题：第一，效率问题，减少人工反复沟通和手工登录平台操作；第二，安全问题，业务人员不直接接触域名账号和 API Key，敏感配置还需要超管飞书确认；第三，管理问题，域名、DNS、申请、审批、到期、日志都有统一记录，方便追踪和复盘。

如果一个团队只有一两个域名，变化很少，那这个系统可能不是刚需。但如果有多个产品、多个域名、多个账号，DNS 变更频繁，并且希望流程规范、权限清晰、记录完整，那这个系统就很适合。

## 14. 给客户或同事演示时怎么讲

演示时建议按“一个真实案例”来讲。

### 演示案例：申请一个新域名并配置解析

可以这样讲：

1. 假设业务要上线一个新产品，需要注册一个域名。
2. 业务同事不用登录后台，只在飞书里提交域名。
3. 系统先判断这个人有没有权限、归属哪个专员。
4. 系统查询域名能不能注册。
5. 域名专员收到飞书审批卡片。
6. 专员确认后，系统自动调用 Cloudflare 或 GoDaddy 注册。
7. 注册完成后，系统保存域名信息。
8. 如果还需要解析，业务继续提交 DNS 记录。
9. 专员审批后，系统自动写入 DNS。
10. 以后在 Web 后台可以查到这个域名、解析记录、申请记录、审批记录和操作日志。

演示重点不要放在按钮多不多，而是放在：

- 业务不用接触敏感账号
- 专员不用到处翻聊天记录
- 超管能控制关键配置
- 每一步都有记录
- 到期和失败能提醒

## 15. 常见问题

### Q1：这个系统是不是替代 Cloudflare？

不是。

Cloudflare、GoDaddy 仍然是实际的域名和 DNS 服务平台。域名管家是内部管理和审批层，负责把公司内部流程和这些平台连接起来。

### Q2：业务人员能不能直接改 DNS？

不能。

业务人员只能提交申请。是否执行，由域名专员审批，系统执行。

### Q3：admin 是不是最大权限？

不是。

admin 是系统管理员，主要管用户和系统维护。最高权限是 super_admin。

### Q4：为什么超管自己操作也要飞书确认？

因为确认动作不放在 Web 里，而是放在飞书客户端里。这样即使 Web 会话出问题，攻击者也不能直接完成敏感操作。

### Q5：Web 后台能不能直接审批？

按当前设计，不应该。

Web 后台主要是配置、查询、统计、审计。审批、确认、执行应在飞书侧完成。

### Q6：系统能不能自动购买所有后缀的域名？

不一定。

是否能自动购买，取决于注册商 API 是否支持对应后缀。有些后缀可能在平台注册页面可以手动买，但 API 不支持自动购买。这种情况需要换支持 API 的注册商，或人工购买后再录入系统。

### Q7：如果 DNS 执行失败怎么办？

系统会记录失败状态和失败原因。业务人员通常只需要看到失败结果；域名专员可以看到更详细原因，用于排查，比如 API 权限不足、记录冲突、网络超时等。

### Q8：系统是否会保存 API Key？

会保存服务商账号所需的密钥，但必须加密存储，并且不能硬编码在代码里。新增或修改这类敏感配置必须走超管飞书确认。

## 17. 项目结构给非技术人员怎么理解

不用记代码目录，只需要知道有几块：

| 部分 | 类比 | 作用 |
|---|---|---|
| 后端 backend | 大脑和执行器 | 判断权限、处理申请、调用外部平台、保存数据 |
| 前端 frontend | 管理后台 | 给专员、管理员、超管查询和配置 |
| 数据库 | 账本 | 保存用户、域名、申请、DNS、日志 |
| 飞书集成 | 办公入口 | 提交申请、审批、通知、超管确认 |
| 服务商适配器 | 外部平台连接器 | 对接 Cloudflare、GoDaddy 等 |
| 定时任务 | 闹钟 | 检查域名到期、同步状态 |
| Nginx / Docker | 部署底座 | 让系统稳定运行在服务器上 |

## 18. 这个项目的核心卖点

可以总结成五点：

1. 流程规范：域名和 DNS 需求不再靠聊天记录推进。
2. 权限清晰：业务、专员、管理员、超管职责分离。
3. 安全可控：API Key 加密，敏感操作需要飞书二次确认。
4. 自动执行：对接 Cloudflare、GoDaddy，减少人工登录平台操作。
5. 可追踪：申请、审批、执行、失败原因、操作日志都有记录。

## 19. 最容易讲错的地方

介绍时注意避免这些误解：

1. 不要说“所有人都可以在后台改 DNS”。正确说法是：业务在飞书申请，专员审批，系统执行。
2. 不要说“Web 后台负责审批”。正确说法是：审批和确认主要在飞书完成。
3. 不要说“admin 什么都能管”。正确说法是：admin 管系统和人员，不参与域名业务；super_admin 才是最高权限。
4. 不要说“系统能自动买所有域名”。正确说法是：能否自动购买取决于注册商 API 能力。
5. 不要说“飞书确认可以省略”。正确说法是：敏感操作必须飞书二次确认。

## 20. 最短总结

域名管家适合域名多、DNS 变更多、需要审批和安全控制的公司。

它把“谁要域名、谁审批、谁执行、执行结果如何、以后怎么查”这件事统一管起来。

业务同事用飞书提需求，域名专员用飞书审批，系统自动对接外部平台执行，Web 后台负责配置、查询、统计和审计。

它的本质不是“域名列表”，而是“域名和 DNS 的内部流程管理系统”。
