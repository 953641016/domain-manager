## 1. 角色定义

| 角色 | 标识 | Web 登录 | 说明 |
|------|------|---------|------|
| 业务人员 | `business` | ❌ | 通过飞书发起域名/DNS 申请 |
| 域名专员 | `domain_spec` | ✅ | 管理分配给自己的域名账号，审批业务人员申请 |
| 系统管理员 | `admin` | ✅ | 管理人员（用户），**不涉及域名账号** |
| 超级管理员 | `super_admin` | ✅ | 唯一，拥有所有权限，负责授权确认 |

> **职责分离原则**：  
> - `admin` 管人，不管账号  
> - `domain_spec` 管账号，不管人  
> - `super_admin` 是两条授权链的终点

## 3. 飞书授权确认流程

### 3.0 核心安全原则：飞书客户端是唯一确认闸门（带外确认）

> **所有敏感写操作，无论由谁发起（包括超级管理员本人），其最终执行权都收口到
> 超级管理员的飞书客户端确认。**

**为什么这样设计：**
- Web 端是攻击面最大的入口（XSS、会话劫持、CSRF、内网渗透）。
- 把"是否真正执行"的开关放到超管手机飞书客户端上，相当于引入一个**带外的第二信道**。
- 即使 Web 后台会话被攻破，攻击者没有超管的飞书，也无法完成任何敏感写操作。
- 飞书客户端确认 ≈ 事实上的二次因子认证（2FA）。

**由此带来的模型简化：**
- 不区分"谁发起"，统一为一条链路：`发起 → 建确认记录 → 超管飞书确认 → 执行`。
- **审批人恒为超级管理员**（不再有"禁止自批"概念——本设计的安全性来自带外信道，而非双人复核）。

### 3.1 统一确认流程

```
发起人（专员 / 管理员 / 超管本人）在 Web 后台操作
        ↓
系统创建 Confirmation 记录（status: pending, expires_at = now+24h）
   ⚠️ 创建前校验：资源归属、操作合法性
        ↓
系统向【超级管理员】飞书客户端发送确认卡片
  ┌────────────────────────────┐
  │ 🔐 操作确认                 │
  │ 操作人：张三（域名专员）     │
  │ 操作类型：新增注册账号       │
  │ 账号名：Cloudflare-主账号   │
  │ API Key：****5678           │
  │  [✅ 确认执行]  [❌ 拒绝]   │
  └────────────────────────────┘
        ↓ 超管在飞书点击（带外确认）
系统校验：点击人 == 超级管理员？（否则拒绝）
        ↓ 是
系统执行实际操作（事务内写库）
        ↓
写审计日志 + 通知发起人（成功 / 拒绝原因）
```

> 当发起人本身就是超管时，卡片同样发到超管自己的飞书，由超管在**手机端**确认——
> 这正是带外确认的价值：Web 端发起、移动端放行，双信道缺一不可。

### 3.2 确认记录字段说明

| 字段 | 说明 |
|------|------|
| `operation_type` | 操作类型（见下方枚举） |
| `initiator_user_id` | 发起人 |
| `source` | 发起来源：`web` / `feishu_doc` / `feishu_client` / `feishu_bot` |
| `requires_super_admin` | 是否需要超管确认（所有需确认操作均为 true） |
| `operation_details` | 操作载荷（JSON，待执行的数据；敏感字段打码/加密） |
| `status` | pending / approved / rejected / cancelled / expired |
| `execution_status` | 执行结果：pending / success / failed |
| `feishu_message_id` | 绑定飞书卡片消息，防重复处理（幂等） |
| `expires_at` | 确认有效期（创建时间 + 24h） |
| `approver_user_id` | 确认人（恒为超管） |
| `confirmed_at` | 确认时间 |

### 3.3 操作留痕（审计要求）⭐

> **每一次敏感操作，从申请到执行，全链路必须留痕，可追溯。**

每条审计记录必须包含：

| 维度 | 记录内容 |
|------|---------|
| **谁申请** | 发起人 ID、姓名、角色 |
| **从哪发起** | 来源：Web 后台 / 飞书文档 / 飞书客户端 / 飞书机器人 |
| **申请什么** | 操作类型 + 操作对象 + 变更内容（敏感字段打码） |
| **谁授权** | 确认人 ID、姓名（恒为超管） |
| **怎么授权** | 确认渠道：飞书客户端（带外确认） |
| **何时** | 申请时间、确认时间 |
| **结果** | 通过 / 拒绝（含原因）/ 超时作废；执行成功 / 失败（含错误） |
| **来源信息** | 发起请求的 IP、User-Agent（Web 端） |

写入时机：
- 创建确认记录时 → 写一条 `xxx_requested`（申请已提交）
- 超管确认/拒绝时 → 写一条 `xxx_approved` / `xxx_rejected`
- 执行完成时 → 写一条 `xxx_executed`（成功/失败）

> 实现：复用现有 `AuditLog` 模型；`source` / `approver` 等维度若现有列不足，
> 存入 `after_state` JSON 或补充列。

### 3.4 操作类型枚举

```
域名账号类：
  add_reg_account      新增注册账号
  update_reg_account   修改注册账号
  delete_reg_account   删除注册账号
  add_dns_account      新增解析账号
  update_dns_account   修改解析账号
  delete_dns_account   删除解析账号
  set_default_config   设置默认配置

人员管理类：
  add_user             添加用户
  update_user          修改用户信息
  update_user_role     变更用户角色
  delete_user          删除/禁用用户
  set_specialist       设置归属专员
```

### 3.5 确认超时处理

| 规则 | 值 |
|------|----|
| 确认有效期 | 24 小时 |
| 超时自动作废 | 状态变为 `cancelled` |
| 超时通知 | 通知发起人重新提交 |

## 4. 数据隔离规则

### 4.1 域名专员视角

- **只能看到** `owner_id = 自己` 的注册账号、解析账号、域名
- **只能审批** `assigned_specialist_id = 自己` 的业务人员申请
- 无法访问其他专员的任何数据

### 4.2 系统管理员视角

- 可以看到**所有用户**，但**看不到**注册账号、解析账号
- 可以看到**所有申请记录**（用于管理追踪）
- 无法访问域名列表中的账号信息

### 4.3 超级管理员视角

- 可以看到所有数据，无隔离限制

### 4.4 业务人员→专员路由

```
业务人员（business）
  └── assigned_specialist_id  →  域名专员
        ↑ 由 admin 或 super_admin 在用户管理中设置
```

业务人员提交申请时：
- 系统自动查找 `requester.assigned_specialist_id`
- 将审批通知发送给对应专员
- 若未设置归属专员，发给所有专员或系统管理员

## 5. 人员添加与权限设置流程

### 5.1 添加业务人员（推荐：扫码添加）

```
管理员在后台点击"扫码添加"
        ↓
生成飞书 OAuth 二维码
        ↓
新员工用飞书扫码
        ↓
系统自动获取飞书用户信息
        ↓
创建用户（默认角色：business）
        ↓
管理员设置归属专员（assigned_specialist_id）⚠️ 需超管确认
```

### 5.2 添加域名专员

```
管理员（admin）在后台创建用户
  └─ 角色选择：domain_spec
  └─ 飞书用户 ID：填写或搜索
        ↓ 提交
系统向超管发飞书确认卡片
        ↓ 超管确认
用户创建成功，通知管理员
```

### 5.3 变更用户角色

所有角色变更均需超管飞书确认，记录变更前后的角色值到审计日志。

### 5.4 禁用/删除用户

- **禁用**（推荐）：保留数据，仅阻止登录
- **删除**：高危操作，需超管确认，保留申请记录（软删除）

## 7. 工程评审结论与修正计划

> 设计骨架（确认流 + owner_id 隔离 + 职责分离 + 带外确认）已确认可行。
> 以下为落地前必须解决的工程项，按优先级排列。

### 7.1 🔴 必须先做（否则有越权/安全漏洞）

| # | 问题 | 修正 |
|---|------|------|
| 1 | 权限定义三处矛盾（permission.py / 文档 / 硬编码角色判断） | 确立 `ROLE_PERMISSIONS` 为唯一事实源；API 改用 `require_permission`，删除硬编码 `role == "xxx"`；修正 admin 的 `can_config_accounts` 为 False |
| 2 | 专员可操作他人账号（无归属校验） | 创建确认前校验 `account.owner_id == 发起人` |
| 3 | API Key 明文存入确认记录 + 账号表 | 确认记录中的 key 加密；最终落库启用 `encryption.py` |
| 4 | "禁止自批"逻辑挡死超管自确认 | 改为「审批人必须 == 超级管理员」，废弃 initiator≠approver 检查 |

### 7.2 🟠 可用性与健壮性

| # | 问题 | 修正 |
|---|------|------|
| 5 | 超管单点（Bus Factor=1），故障即冻结全部写操作 | 支持配置≥1 个超管；确认记录加超时兜底；考虑超管不可用时的降级策略 |
| 6 | 模型缺 `expires_at`，24h 超时无处落地 | 加 `expires_at` 字段 + 定时作废任务 |
| 7 | 飞书卡片可重复点击 / webhook 重投 → 重复执行 | 用 `feishu_message_id` + 状态机做幂等，执行与状态更新包在同一事务 |
| 8 | 执行失败但状态已是 approved（状态不一致） | 加 `execution_status`，失败标记并如实通知，不静默吞异常 |

### 7.3 🟡 设计完整性

| # | 问题 | 修正 |
|---|------|------|
| 9 | TOCTOU：批准时资源可能已变 | 执行前重新校验资源存在性与归属 |
| 10 | 全链路强依赖飞书，Web 看不到状态 | Web 后台加"待我审批"列表作兜底入口 |
| 11 | 敏感操作无审计日志 | 确认执行后统一写 `AuditLog` |
| 12 | admin 仍能调账号接口 | 按 ROLE_PERMISSIONS 收口 |

### 7.4 修正实施顺序

```
第一阶段（地基）：  #1 统一权限源 → #2 归属校验 → #4 审批人规则
第二阶段（健壮）：  #6 expires_at → #7 幂等 → #8 执行状态 → #11 审计
```
