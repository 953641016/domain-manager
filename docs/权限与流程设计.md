## 0. 架构分层总纲（最高原则）

> **系统分为两层，职责严格分离。其他所有规则都从此派生。**

### 业务层 —— 飞书驱动
- **内容**：域名申请、域名注册、解析申请、DNS 解析、申请审批
- **入口**：飞书多维表格（文档）+ 飞书客户端/机器人
- **特点**：所有业务流程的发起、审批、确认、执行**都在飞书完成**
- **Web 不参与业务流程**

### 配置/管理层 —— Web 后台
- **内容**：注册商/解析商配置、域名账号配置、默认配置、人员管理、权限管理、
  记录查询、统计与日志
- **入口**：Web 后台
- **特点**：是业务层**之上**的配置与沉淀，**只做配置和查询，不推进业务**

### 两层的交汇点
- Web 配置好的账号/服务商/默认值/人员归属 → 供飞书侧业务流程使用
- 飞书侧业务流程产生的记录 → 沉淀到库，供 Web 查询审计
- **敏感写操作（账号/服务商/人员）虽在 Web 发起，但执行权收口到飞书超管确认**（见 3.0 带外确认）

## 1. 角色定义

| 角色 | 标识 | Web 登录 | 说明 |
|------|------|---------|------|
| 业务人员 | `business` | ❌ | 通过飞书发起域名/DNS 申请；**必须归属且仅归属一个专员**，否则无权申请 |
| 域名专员 | `domain_spec` | ✅ | 管理分配给自己的域名账号、审批业务人员申请、**可在飞书直接注册/解析**（免审批） |
| 系统管理员 | `admin` | ✅ | **技术运维/迭代岗**：管理人员（用户）+ 系统维护；无域名账号权限，**无任何业务侧（注册/解析）权限** |
| 超级管理员 | `super_admin` | ✅ | 唯一，**技术岗 + 业务岗双重身份**：既可管人/管系统，也可做业务（注册/解析）；同时是所有授权确认的唯一终点 |

> **职责分离原则**：
> - `admin` = 技术运维岗，**只管人、管系统**，不碰账号、不碰业务流程（为后续项目迭代/维护而设）
> - `domain_spec` = 业务岗，管自己的账号、做业务（注册/解析/审批），**不管人**
> - `super_admin` = **技术 + 业务双岗**，拥有全部权限，且是唯一的飞书确认闸门
> - 业务（注册/解析）能力归属于 `business / domain_spec / super_admin`，**`admin` 不参与业务**
> - 人员管理能力归属于 `admin / super_admin`，**`domain_spec` 不管人**

> ⚠️ **Web 无确认权（对所有人，含超管）**：
> Web 上任何角色都**没有「确认/授权」这个动作**——人员类、业务（域名）类敏感操作的
> 确认权**只存在于超管的飞书客户端**。超管即便在 Web 发起操作，也必须切到飞书客户端确认。
> （详见 3.0 带外确认）

### 2.4 Web 后台定位（重要）

> **Web 后台 = 业务侧之上的「配置」+ 业务侧「记录查询」。Web 不感知、不参与业务流程。**

- ✅ Web 负责：账号/默认配置、人员配置、记录与日志查询、统计
- ❌ Web 不负责：审批、确认、业务流程推进——这些**全部在飞书完成**
- Web 上的"待审批/待确认"等列表**只读展示**，无任何确认/执行按钮
- 业务流程（申请→审批→执行→通知）的交互入口是飞书机器人/卡片，不是 Web

> 含义：现有 Web 端比需求复杂，后续应**做减法**——剥离业务流程参与能力，回归"配置 + 查询"。

### 2.5 Web 端各角色权限边界（清晰划线）

> **① admin 不碰业务　② domain_spec 不碰人。**

## 3. 飞书授权确认流程

### 3.0 核心安全原则：飞书客户端是唯一确认闸门（带外确认）

> **所有敏感写操作，无论由谁发起（包括超级管理员本人），其最终执行权都收口到
> 超级管理员的飞书客户端确认。**

**为什么这样设计：**
- Web 端是攻击面最大的入口（XSS、会话劫持、CSRF、内网渗透）。
- 把"是否真正执行"的开关放到超管手机飞书客户端上，相当于引入一个**带外的第二信道**。
- 即使 Web 后台会话被攻破，攻击者没有超管的飞书，也无法完成任何敏感写操作。
- 飞书客户端确认 ≈ 事实上的二次因子认证（2FA）。

**由此带来的模型简化：**
- 不区分"谁发起"，统一为一条链路：`发起 → 建确认记录 → 超管飞书确认 → 执行`。
- **审批人恒为超级管理员**（不再有"禁止自批"概念——本设计的安全性来自带外信道，而非双人复核）。

### 3.1 统一确认流程

```
发起人（专员 / 管理员 / 超管本人）在 Web 后台操作
        ↓
系统创建 Confirmation 记录（status: pending, expires_at = now+24h）
   ⚠️ 创建前校验：资源归属、操作合法性
        ↓
系统向【超级管理员】飞书客户端发送确认卡片
  ┌────────────────────────────┐
  │ 🔐 操作确认                 │
  │ 操作人：张三（域名专员）     │
  │ 操作类型：新增注册账号       │
  │ 账号名：Cloudflare-主账号   │
  │ API Key：****5678           │
  │  [✅ 确认执行]  [❌ 拒绝]   │
  └────────────────────────────┘
        ↓ 超管在飞书点击（带外确认）
系统校验：点击人 == 超级管理员？（否则拒绝）
        ↓ 是
系统执行实际操作（事务内写库）
        ↓
写审计日志 + 通知发起人（成功 / 拒绝原因）
```

> 当发起人本身就是超管时，卡片同样发到超管自己的飞书，由超管在**手机端**确认——
> 这正是带外确认的价值：Web 端发起、移动端放行，双信道缺一不可。

### 3.2 确认记录字段说明

| 字段 | 说明 |
|------|------|
| `operation_type` | 操作类型（见下方枚举） |
| `initiator_user_id` | 发起人 |
| `source` | 发起来源：`web` / `feishu_doc` / `feishu_client` / `feishu_bot` |
| `requires_super_admin` | 是否需要超管确认（所有需确认操作均为 true） |
| `operation_details` | 操作载荷（JSON，待执行的数据；敏感字段打码/加密） |
| `status` | pending / approved / rejected / cancelled / expired |
| `execution_status` | 执行结果：pending / success / failed |
| `feishu_message_id` | 绑定飞书卡片消息，防重复处理（幂等） |
| `expires_at` | 确认有效期（创建时间 + 24h） |
| `approver_user_id` | 确认人（恒为超管） |
| `confirmed_at` | 确认时间 |

### 3.3 操作留痕（审计要求）⭐

> **每一次敏感操作，从申请到执行，全链路必须留痕，可追溯。**

每条审计记录必须包含：

| 维度 | 记录内容 |
|------|---------|
| **谁申请** | 发起人 ID、姓名、角色 |
| **从哪发起** | 来源：Web 后台 / 飞书文档 / 飞书客户端 / 飞书机器人 |
| **申请什么** | 操作类型 + 操作对象 + 变更内容（敏感字段打码） |
| **谁授权** | 确认人 ID、姓名（恒为超管） |
| **怎么授权** | 确认渠道：飞书客户端（带外确认） |
| **何时** | 申请时间、确认时间 |
| **结果** | 通过 / 拒绝（含原因）/ 超时作废；执行成功 / 失败（含错误） |
| **来源信息** | 发起请求的 IP、User-Agent（Web 端） |

写入时机：
- 创建确认记录时 → 写一条 `xxx_requested`（申请已提交）
- 超管确认/拒绝时 → 写一条 `xxx_approved` / `xxx_rejected`
- 执行完成时 → 写一条 `xxx_executed`（成功/失败）

> 实现：复用现有 `AuditLog` 模型；`source` / `approver` 等维度若现有列不足，
> 存入 `after_state` JSON 或补充列。

### 3.4 操作类型枚举

```
域名账号类：
  add_reg_account      新增注册账号
  update_reg_account   修改注册账号
  delete_reg_account   删除注册账号
  add_dns_account      新增解析账号
  update_dns_account   修改解析账号
  delete_dns_account   删除解析账号
  set_default_config   设置默认配置

人员管理类：
  add_user             添加用户
  update_user          修改用户信息
  update_user_role     变更用户角色
  delete_user          删除/禁用用户
  set_specialist       设置归属专员
```

### 3.5 确认超时处理

| 规则 | 值 |
|------|----|
| 确认有效期 | 24 小时 |
| 超时自动作废 | 状态变为 `cancelled` |
| 超时通知 | 通知发起人重新提交 |

## 4. 数据隔离规则

### 4.1 域名专员视角

- **只能看到** `owner_id = 自己` 的注册账号、解析账号、域名
- **只能审批** `assigned_specialist_id = 自己` 的业务人员申请
- 无法访问其他专员的任何数据

### 4.2 系统管理员视角

- 可以看到**所有用户**，但**看不到**注册账号、解析账号
- 可以看到**所有申请记录**（用于管理追踪）
- 无法访问域名列表中的账号信息

### 4.3 超级管理员视角

- 可以看到所有数据，无隔离限制

### 4.4 业务人员→专员路由

```
业务人员（business）
  └── assigned_specialist_id  →  域名专员
        ↑ 由 admin 或 super_admin 在用户管理中设置
```

业务人员提交申请时：
- 系统自动查找 `requester.assigned_specialist_id`
- 将审批通知发送给对应专员
- **若未设置归属专员，直接拒绝申请并提示联系管理员配置**（见 ADR-3，不存在兜底分支）

## 5. 人员添加与权限设置流程

### 5.1 添加业务人员（推荐：扫码添加）

```
管理员在后台点击"扫码添加"
        ↓
生成飞书 OAuth 二维码
        ↓
新员工用飞书扫码
        ↓
系统自动获取飞书用户信息
        ↓
创建用户（默认角色：business）
        ↓
管理员设置归属专员（assigned_specialist_id）⚠️ 需超管确认
```

### 5.2 添加域名专员

```
管理员（admin）在后台创建用户
  └─ 角色选择：domain_spec
  └─ 飞书用户 ID：填写或搜索
        ↓ 提交
系统向超管发飞书确认卡片
        ↓ 超管确认
用户创建成功，通知管理员
```

### 5.3 变更用户角色

所有角色变更均需超管飞书确认，记录变更前后的角色值到审计日志。

### 5.4 禁用/删除用户

- **禁用**（推荐）：保留数据，仅阻止登录
- **删除**：高危操作，需超管确认，保留申请记录（软删除）

## 8. 关键设计决策记录（ADR）

> 以下为已拍板的设计决策，含委托给工程方做出的决策。

### ADR-1：超级管理员唯一，接受单点风险 ✅
- **决策**：不引入备用超管。超管唯一。
- **理由**：本系统是对既有人工流程的**自动化增强**，不是唯一通道。紧急情况下，
  域名专员可直接登录注册商/解析商官方平台手工操作，**绕过本系统**——天然的逃生通道。
- **影响**：超管不可用时系统写操作冻结属可接受；但 `expires_at` 超时兜底仍需实现，
  避免记录永久 pending。

### ADR-2：Web 不可确认，仅飞书客户端确认 ✅
- **决策**：确认/执行只能在飞书客户端完成；Web 无确认入口。
- **理由**：带外确认（见 3.0）。Web 是高风险攻击面，不应持有敏感操作的执行权。
- **影响**：Web 重新定位为"配置 + 查询"（见 2.4），需做减法剥离业务流程参与。

### ADR-3：业务人员必须归属唯一专员 ✅
- **决策**：`business` 用户必须设置 `assigned_specialist_id`，且只能归属一个专员；
  未归属者无权发起申请。
- **影响**：申请创建前校验归属；不存在"发给所有专员"的兜底分支（更简单、更明确）。

### ADR-4：admin 不参与业务侧 ✅
- **决策**：`admin` 仅管理人员；无注册/解析/审批/账号权限。
- **影响**：需修正 `permission.py`——admin 的 `can_submit_register / can_direct_register /
  can_renew / can_approve / can_config_accounts` 全部改为 False；API 按权限收口。

### ADR-5：数据库迁移方案（工程决策）✅
- **决策**：**不引入 Alembic**，采用轻量幂等迁移脚本 `backend/scripts/migrate.py`。
- **做法**：
  - 用 `PRAGMA table_info` 检测缺失列，按需 `ALTER TABLE ADD COLUMN`（SQLite 支持）；
  - 新表交给 `Base.metadata.create_all` 自动创建；
  - 脚本幂等，可重复执行；在 backend 启动时或 `deploy.sh` 中自动运行。
- **理由**：单库、小规模、SQLite，Alembic 属过度工程；轻量脚本符合项目简单优先的取向。
- **未来**：当 schema 稳定 / 团队扩大 / 切换 Postgres 时，再迁移到 Alembic。
- **本次需补的列**：
  - `users.assigned_specialist_id`
  - `user_operation_confirmations`: `source` / `execution_status` / `expires_at`
  - 账号表的 `api_key / api_secret` 改存密文（值迁移，见 ADR-6）

### ADR-6：敏感凭证加密方案（工程决策）✅
- **决策**：API Key / Secret 用 **Fernet 对称加密**（复用 `core/encryption.py`）。
- **密钥管理**：
  - `ENCRYPTION_KEY`（32 字节 urlsafe base64）存于服务器 `.env`，不进 Git；
  - 首次生成后**安全备份**——密钥丢失 = 所有已存凭证不可解密。
- **读写规则**：
  - 写入：Service 层加密后落库；
  - 读取：仅在调用注册商/解析适配器时解密，**绝不返回给前端**；
  - 展示：API 只返回打码值（`****1234`）或不返回；
  - 确认记录 `operation_details` 中的 key **存密文**，飞书卡片只显示打码值。
- **存量迁移**：迁移脚本对现有明文 key 加密（幂等：先尝试解密，失败即视为明文再加密）。

### ADR-7：留痕/审计 ✅
- 见 3.3。每次敏感操作记录：谁申请、从哪发起（web/feishu_doc/feishu_client/feishu_bot）、
  申请内容、谁授权、确认渠道、时间、结果。三个写入时机：requested / approved-rejected / executed。

### ADR-8：服务商（注册商/DNS）改为可管理的数据库表 ✅
- **决策**：注册商/解析商从硬编码工厂类改为数据库表，支持增删改。
- **权限**：与账号管理一致——`domain_spec` / `super_admin` 可操作，**经超管飞书确认**；
  `admin` 无权。
- **影响**：
  - 新增 `registrars` / `dns_providers` 表（替代 `RegistrarFactory` 的硬编码列表）；
  - 适配器工厂改为按 `code` 从库读取启用的服务商；
  - 新增操作类型：`add_provider` / `update_provider` / `delete_provider`；
  - ⚠️ 删除服务商需校验是否仍有账号/域名引用，有引用则禁止删除。

### ADR-9：飞书文档（多维表格）发起 ✅ 分步实现
- **决策**：支持从飞书多维表格发起申请（`source=feishu_doc`），**分步实现**。
- **分步**：
  1. 先打通飞书机器人/客户端发起（`source=feishu_client/feishu_bot`）；
  2. 再接入飞书多维表格按钮 → 工作流 → 调用 `POST /feishu/request`；
  3. 申请状态回写飞书表格。
- **影响**：本期先保证 source 字段贯通与留痕，多维表格集成作为后续步骤。

```
add_provider     新增服务商（注册商/解析商）
update_provider  修改服务商
delete_provider  删除服务商
```
